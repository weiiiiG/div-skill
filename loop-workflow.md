# 分轮开发工作流

> 每轮只做一件事，做完就测，测完再下一轮。以任务完成为轮次推进标准，而非时间间隔。

## 一句话模板

```
搭最小可行版本的[核心功能/骨架]，每完成一个功能就自测（[具体测试方式]），
发现问题当场修再继续下一轮，全部完成后做一轮完整的全面验收
```

## 各场景的现成提示词

### 前端页面

```
先搭HTML结构和容器框架，每完成一个组件就自测（打开预览确认布局无溢出、交互无报错），
修完bug再进入下一个组件，全部完成后做完整验收
```

### React 组件

```
先实现最小可用版本，每完成一个文件就检查编译无报错，每个组件渲染后自测交互状态，
问题及时修再继续，完成全部功能后做全面验收
```

### API / 后端

```
先搭路由和数据模型骨架，每实现一个接口就curl测试，确认返回正确再进入下一个，
全部完成后跑整套测试做验收
```

### 重构优化

```
先摸清当前代码结构和数据流，每重构完一个模块就对比改前改后效果，
验证功能一致再继续下一个，全部完成后做整体验收
```

## 核心原则

| 原则 | 说明 |
|---|---|
| **最小可行版本** | 第一轮只搭建最核心的功能骨架，不追求完美 |
| **完成一个测一个** | 每轮结束前必须自测，确认当前功能可用 |
| **问题当场修** | 测试发现的问题在本轮内修完，不留到下一轮 |
| **无阻塞再推进** | 本轮功能通过自测后，才进入下一轮开发 |
| **最终完整验收** | 全部功能完成后，做一次全面的端到端验收 |

## 推进方式

以任务完成为轮次标准，而非时间：

```
Round 1: 搭骨架 → 自测通过 → 手动发起下一轮
Round 2: 加功能A → 自测发现bug → 修bug → 复测通过 → 下一轮
Round 3: 加功能B → 自测通过 → 下一轮
...
Final:   完整验收 → 全部通过 → 交付
```

每轮结束后在当前回复中说明完成情况和自测结果，等待用户确认后进入下一轮。

## 验收清单

- [ ] 核心功能是否正常工作
- [ ] 是否有报错/警告
- [ ] 边缘情况是否处理
- [ ] 改前改后效果是否一致（重构类）
- [ ] 所有自测用例是否通过
